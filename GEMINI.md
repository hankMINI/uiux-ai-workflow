# UI/UX AI 工作流程記憶 (GEMINI.md)

## 1. 專案目錄架構
- **01_核心戰略與方法論**: SOP 與轉型提案
- **02_實戰專案案例**: 實際專案檔案 (例：龍山寺)
- **03_設計系統產出**: Token, Prompts, Guideline
- **04_分享與報告**: HTML 成果展示
- **05_系統設定與技能庫**: MCP 與 Skill 設定檔

## 2. 核心 AI 工作流程 (已驗證)
1. **搜尋亮點 (策略)**: NotebookLM (海量資料分析)
2. **提案簡報 (排版)**: Markdown 結構流 -> Google Slides
3. **人天預估 (報價)**: Project Architect Skill (WBS 輸出)
4. **建立規範 (視覺)**: Design System Gen + Figma AI
5. **高擬真交付 (確認)**: Figma Make AI (直接生成畫面)
6. **UIUX Flow (開發)**: Pencil.dev + Claude MCP (畫完即開發)

## 3. 報告規劃
- **內容來源**: `04_分享與報告/WORKFLOW_CONTENT.md` (Markdown 核心)
- **展示輸出**: `04_分享與報告/UIUX_WORKFLOW_SHARE.html` (自動同步更新)
- **製作進度**: 
    - 已完成：Header, 痛點 1-5, 方法論, 回應痛點 1 (雙軌調研)。
    - 待續：回應痛點 2 (簡報排版), 痛點 3 (報價), 痛點 4 (DS), 痛點 5 (高擬真交付)。

## 4. 使用者偏好
- 回答必須精簡、直接給重點，拒絕廢話。
- 嚴格遵守台灣繁體中文 UI/UX 用語。
- 禁止使用「工作流」，一律使用「工作流程」。
- 禁止在中文旁使用括號標註英文（例如：禁止「解決方案 (Solution)」）。
- 視覺規範: 不使用 emoji，改用 SVG icon 呈現（符合 EY 專業風格）。
- 文案角色: 定位為「安永 (EY) 顧問級別的簡報策略師」。
    - 邏輯結構：問題 (Pain) -> 方案 (Solution) -> 價值 (ROI)。
- 文字規範: 全面禁用雙星號標記 (bolding)。文字力求真實直白，去除多餘形容詞。
